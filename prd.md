# 版本变更管理系统 — 产品需求文档 (PRD)

> **版本**: v2.0 | **日期**: 2026-06-04 | **状态**: 评审中

---

## 1. 文档概述

### 1.1 版本记录

| 版本 | 日期 | 变更说明 |
|------|------|----------|
| v1.0 | 2026-06-04 | 初始版本：变更管理、多数据库、用户系统、AI 能力 |
| v1.1 | 2026-06-04 | 组织结构、细粒度权限、DB 导出、AI 说明书、文档自动分析 |
| v1.2 | 2026-06-04 | AI 模型/Skill/MCP 管理 |
| **v2.0** | 2026-06-04 | **Git 集成、Jenkins 构建、技能体系、Agent 权限策略** |

### 1.2 项目背景

当前版本变更管理存在以下痛点：

- 图片/截图散乱存储，无法统一检索
- 变更与需求、故障之间缺乏关联追溯
- 变更影响范围依赖人工判断，易遗漏
- 历史变更经验无法沉淀复用
- 单一数据库限制，无法适配不同业务线的数据库环境
- 缺乏用户个性化能力（头像、微信快捷登录）
- 权限粒度粗糙，无法区分查看/编辑/审批
- 缺乏 DB 变更记录的批量导出能力
- 变更说明书依赖人工编写，效率低
- 变更录入依赖人工填写，易遗漏
- AI 能力缺乏统一管理（模型/Skill/MCP）
- **变更与 Git 代码库无关联，无法追溯代码变更**
- **缺乏自动化构建能力，版本发布依赖人工操作**

### 1.3 目标

构建一套**开源、私有化、分阶段落地**的版本变更管理系统：

| 阶段 | 时间 | 目标 |
|------|------|------|
| **MVP** | 第 1-2 周 | 变更 CRUD、组织架构、用户系统、Git 关联、Jenkins 触发 |
| **Phase 2** | 第 3-4 周 | AI-RAG 检索、GraphRAG 图谱、文档自动分析、MCP 管理 |
| **Phase 3** | 第 5-6 周 | AI 说明书、Skill 管理、多数据库适配 |

### 1.4 技术选型

| 组件 | 选型 | 协议 | 说明 |
|------|------|------|------|
| 前端 | 自研 React/Vue | - | 不依赖 OpenProject 前端 |
| 后端 | Spring Boot / Node.js | - | REST API |
| 对象存储 | MinIO | Apache2.0 | 图片/附件/头像存储 |
| RAG 引擎 | RAGFlow | Apache2.0 | 文档解析+图片 OCR |
| 知识图谱 | GraphRAG | MIT | 业务链路知识图谱 |
| 数据库 | MySQL（MVP）→ 多库适配 | - | MVP 先用 MySQL |
| 用户认证 | JWT + 微信 OAuth2.0 | - | 账号密码 + 微信扫码 |
| Git 集成 | Git Java / Nodegit | - | 分支管理、合并操作 |
| Jenkins | Jenkins REST API | - | 构建触发、状态回写 |
| MCP 框架 | Model Context Protocol | MIT | 统一 AI 工具接入标准 |
| AI 能力 | LLM + Vision + Skill + MCP | - | 可插拔技能体系 |

### 2.3 技能体系架构

系统将核心能力封装为独立 Skill 模块，通过 Skill 注册表统一管理：

```
skills/
├── README.md                          # 技能注册表
├── git-management/
│   └── SKILL.md                       # Git 管理：仓库、分支、合并
├── jenkins-integration/
│   └── SKILL.md                       # Jenkins 集成：构建、状态、日志
└── agent-policy/
    └── SKILL.md                       # Agent 策略：角色、权限、操作线路
```

**Skill 更新流程：**
1. 修改对应 `SKILL.md` 文件
2. 更新版本号
3. 更新 `skills/README.md` 版本记录
4. Agent 重新加载策略

```
公司

      └── 团队（组）
```

### 3.4 权限边界规则

| 规则 | 说明 |
|------|------|
| **数据隔离** | 用户只能访问已授权团队/部门的数据 |
| **禁止操作** | 不得执行 destructive 操作（force push、删除受保护分支、删除 Job） |
| **敏感信息** | 不得泄露密码、Token、密钥等敏感信息 |
| **操作日志** | 所有操作必须记录审计日志 |
| **Git 保护分支** | main/master/release 分支禁止直接删除，合并需额外审批 |
| **Jenkins 安全** | 不得修改 Jenkins 全局安全配置 |

## 4. 核心业务流程

### 4.1 变更全生命周期流程

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ ①变更申请 │ →  │ ②关联 Git │ →  │ ③刷新分支 │ →  │ ④执行合并 │ →  │ ⑤触发构建 │
│          │    │          │    │          │    │          │    │          │

│影响范围  │    │          │    │          │    │          │    │          │

                                      │               │        │条目状态  │
                                      │               │        └──────────┘
```

### 4.2 管理员操作流程

```
管理员操作：
  ① 创建组织架构（公司 → 部门 → 团队）
  ② 在团队内添加 Git 仓库（URL + Token/SSH）
  ③ 为用户授权可访问的 Git 仓库（用户级授权）
  ④ 配置 Jenkins 实例（URL + Token）
  ⑤ 为 Git 仓库关联 Jenkins Job 模板
```

### 4.3 成员操作流程

```
成员操作：
  ① 登录 → 看到已授权的 Git 项目列表
  ② 创建变更（需求/BUG）→ 勾选关联的 Git 项目
  ③ 点击"刷新分支" → 系统拉取项目所有分支
  ④ 选择源分支 + 目标分支 → 执行 Git 合并
  ⑤ 合并成功 → 自动触发 Jenkins 构建
  ⑥ 合并失败 → 提示原因 → 人工处理
  ⑦ 查看构建状态 → 回写结果到变更条目
```

### 5.7 升级日志

> **注意**：升级日志查看权限：系统管理员、审计员可查看全部；其他角色仅可查看本团队相关升级记录。

#### 5.7.1 功能定义

| 功能 | 说明 | 权限 |
|------|------|------|
| 查看升级历史 | 按时间/版本/状态筛选升级记录 | 系统管理员、审计员、其他角色（本团队） |
| 查看升级详情 | 查看单次升级的完整日志、关联变更、Git Commit、Jenkins 构建 | 系统管理员、审计员、其他角色（本团队） |
| 查看升级对比 | 对比升级前后的版本差异（变更条目、Git Commit） | 系统管理员、审计员 |
| 回滚升级 | 执行版本回滚操作 | 系统管理员 |
| 导出升级日志 | 导出升级记录为 Markdown/Excel | 系统管理员、审计员 |

### 9.2 安全性

| 安全项 | 措施 |
|--------|------|
| **权限控制** | 角色 × 资源 × 操作三级权限检查 |
| **数据隔离** | 用户只能访问已授权团队/部门的数据 |
| **签名 URL** | MinIO 临时签名链接（30 分钟有效） |
| **数据加密** | 敏感字段（密码/Token/密钥）AES256 加密 |
| **操作审计** | 所有操作记录审计日志 |
| **微信安全** | OAuth2.0 标准流程，state 参数防 CSRF |
| **JWT 安全** | 短期 access_token（2h）+ 长期 refresh_token（7d） |
| **Git 安全** | 禁止 force push、删除保护分支 |
| **Jenkins 安全** | 不得修改 Jenkins 全局安全配置 |

1.2 项目背景

当前版本变更管理存在以下痛点：
• 图片/截图散乱存储，无法统一检索
• 变更与需求、故障之间缺乏关联追溯
• 变更影响范围依赖人工判断，易遗漏
• 历史变更经验无法沉淀复用

本系统目标：构建一套开源、私有化、3周MVP落地的版本变更管理系统， 整合 OpenProject MinIO RAGFlow GraphRAG， 实现变更全生命周期管理 + AI智能检索 + 知识图谱推理。
v1.1 新增：多数据库适配（MySQL/PG/Oracle/DM）+ 用户头像自定义 + 微信OAuth2.0登录

1.3 术语表

📦 业务线 (Business Line)
公司级业务分组，如"支付业务线"、"结算业务线"，对应 OpenProject 项目集 (Portfolio)
📁 产品 (Product)
业务线下的具体产品，如"医保结算系统"，对应 OpenProject 项目 (Project)
🏷️ 版本 (Version)
产品的迭代版本，如"v2.3.1"，对应 OpenProject 里程碑 (Milestone)
🔄 变更项 (Change Item)
版本内的具体变更内容，分为5大类：DB/API/Config/Code/Infra
🔗 功能点 (Function Point)
最小业务功能单元，如"医保结算模块"，关联所有变更时间线
🧠 RAGFlow
文档解析 + 图片OCR + 向量入库的 RAG 引擎，支持自然语言问答
🕸️ GraphRAG
构建业务-数据表-接口知识图谱，自动计算变更影响范围
📦 MinIO
对象存储，统一托管所有图片/附件/头像，替代本地磁盘
🗄️ 多数据库适配
统一SQL方言转换层，支持MySQL/PG/Oracle/DM，产品级独立配置
👤 用户头像
MinIO存储用户头像，支持自定义上传和微信自动拉取，自动生成缩略图
📱 微信OAuth2.0
微信扫码登录，获取openid/unionid，自动绑定/创建用户账户
🔐 JWT
JSON Web Token，短期access_token（2h）+ 长期refresh_token（7d）

5.5 GraphRAG知识图谱

构建业务链路知识图谱，实现变更影响自动推理和历史经验复用。

图谱节点类型

Node: BusinessLine # 业务线 Node: Product # 产品 Node: Version # 版本 Node: ChangeItem # 变更项 Node: Table # 数据表 Node: API # 接口 Node: FunctionPoint # 功能点 Node: Incident # 故障 Node: Image # 图片对象

图谱边关系

# 业务关系 BusinessLine -[包含]-> Product Product -[依赖]-> Table Product -[暴露]-> API # 变更关系 Version -[包含]-> ChangeItem ChangeItem -[影响]-> Table ChangeItem -[影响]-> API ChangeItem -[关联]-> FunctionPoint # 历史关系 ChangeItem -[关联]-> Incident # 变更引发的故障 ChangeItem -[引用]-> Image # 变更关联的图片 Incident -[解决于]-> ChangeItem # 故障的修复变更

5.6 自动同步（Flyway + 配置中心）

DB变更和配置变更自动同步至变更清单，减少人工录入，确保变更记录的完整性。

🔧 Flyway DB变更同步
监控 Flyway migration 脚本执行，自动创建变更条目，关联 affected tables，推送 GraphRAG 更新图谱
⚙️ 配置中心变更同步
监听 Nacos/Apollo 配置变更事件，自动创建 Config 类型变更条目，记录变更前后的值差异

AI 对话澄清AI 提取结果不确定，需要用户补充AI 提问 → 用户选择/输入 → AI 自动填充
混合模式部分清晰 + 部分不确定确认清晰项 + 对话澄清不确定项

版本进度、变更完成率、延期率、人员负载热力图。

1设计目标 & 设计原则

★三大设计目标

  • 随叫随到（In-context）：在每个业务页面，AI 都是可一键唤起的浮窗/侧边栏，识别当前页面上下文主动建议。
  • 可编排（Composable）：AI 能力以"场景（Scenario）"为最小单位，业务线/团队/用户三维度绑定，可热插拔。
  • 可观测（Observable）：所有 AI 调用均有日志、可统计成本、可收集反馈做 RLHF。

★五条设计原则

  1. 不替代人：AI 提供建议与摘要，决策与终审仍由人完成。
  2. 权限最小化：AI 调用受 RBAC 控制；不同角色看到的 AI 能力不同。
  3. 数据可溯源：所有 AI 回答必须带引用的原文/链接（"基于以下 3 条变更..."）。
  4. 降级友好：当 RAG/模型不可用时，自动降级为关键词检索 + 模板填充，业务不中断。
  5. 成本可控：按业务线预算限额；超限自动降级到低成本模型。

F2行内 AI 按钮（Inline AI Button）

嵌入到表单字段旁，触发后弹出小气泡/抽屉。例：

✨AI 撰写在变更内容输入框旁
🛡风险检查在 SQL 输入框旁
🧩影响面在变更类型选完后
📝生成 commit在 Git 提交前

F3智能卡片（Insight Card）

出现在列表/详情页的"AI 洞察"区域，通常以折叠面板/抽屉形式展示。

  • Dashboard: "本周团队 X 出现 5 次失败回滚，主要原因是……"
  • 变更详情: "与变更 #123、#456 高度相似，曾引发过生产事故 INC-2025-08-12"
  • 审批页: "建议批准（综合风险评分 28/100，低于阈值 60）"

F4后台 Worker（Background AI Worker）

无需 UI 交互，AI 定时/触发式执行，例如：

  • 每 10 分钟扫描异常行为日志并发送站内信
  • 构建失败时自动分析根因并推送给负责人
  • 每周一 09:00 自动生成上周周报

// 后端伪代码 (FastAPI)
@router.post("/ai/scenario/{code}/invoke")
async def invoke_scenario(code: str, body: ScenarioInvoke, user=Depends(get_current_user)):
    # 1. 解析场景
    scenario = await load_scenario(code)
    # 2. 解析 binding（业务线→团队→用户，按 priority 取最高优先级）
    binding = await resolve_binding(user, body.context)
    # 3. 注入 RAG 上下文
    docs = await rag_search(body.query, top_k=5, filter=binding.filter)
    # 4. 注入 Skill
    skills = await load_skill(binding.skill_ids)
    # 5. 调 LLM
    answer = await llm_call(scenario.system_prompt, body, docs, skills, model=binding.model_id)
    # 6. 落库
    await log_ai_call(user.id, code, body, answer, cost=...)
    return answer

10风险 & 缓解

风险影响缓解措施
AI 回答存在幻觉 高 强制要求回答带"引用"，UI 显眼展示；提供反馈按钮；高风险场景（金融/医疗）必须二次人工确认
外部模型 API 不稳定 中 多模型 fallback；本地 Ollama 兜底；限流 + 降级提示
成本失控 中 业务线 / 用户维度预算；超限降级到低成本模型；按日对账
敏感数据出域 高 外发前 PII 脱敏；金融业务走私有化部署；提供审计日志
用户抵触（担心被替代） 中 对外口径"AI 助手"；保留"忽略建议"选项；正面激励采纳率
RAGFlow / Ollama 部署复杂 中 提供 docker-compose 一键起；演示环境允许无 RAG 跑通（直接走 DB LIKE 检索）
